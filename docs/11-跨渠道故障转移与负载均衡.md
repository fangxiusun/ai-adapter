# 跨渠道故障转移与负载均衡指南

## 概述

ai-adapter 支持跨渠道故障转移（Failover）和负载均衡，当多个渠道配置了相同模型时，系统可以自动在渠道间切换，提高可用性。

## 工作原理

### 两层重试模型

```
请求进入
  │
  ├─ 跨渠道故障转移层
  │   候选渠道按优先级排序: [primary(10), backup(20)]
  │   对每个渠道：
  │     │
  │     ├─ 渠道内 Key 轮转层
  │     │   轮转 Key: [key-1, key-2, key-3]
  │     │   401/429/4xx → 换 Key
  │     │   5xx/连接失败 → 连续 N 次后跳出渠道
  │     │   所有 Key 用尽 → 返回"渠道失败"
  │     │
  │     ← 收到"渠道失败"
  │     → 转移到下一渠道
  │
  ← 所有渠道失败 → 返回最终错误
```

### 错误处理策略

| 状态码 | 渠道内行为 | 跨渠道行为 | 理由 |
|--------|-----------|-----------|------|
| 400 | 立即返回 | ❌ 不转移 | 用户参数错误 |
| 401 | 永久屏蔽 Key + 轮转 | Key 全部试完 → 转移 | Key 问题 |
| 429 | 轮转下一个 Key | Key 全部试完 → 转移 | Key 级限流 |
| 其他 4xx | 轮转下一个 Key | Key 全部试完 → 转移 | 可能是 Key 相关 |
| 5xx | 连续 N 个失败 → 跳出 | 转移 | 服务器故障 |
| 连接失败 | 连续 N 个失败 → 跳出 | 转移 | 服务器不可达 |

### 渠道健康状态

系统自动追踪每个渠道的健康状态：

- **健康**：正常处理请求
- **不健康**：连续 N 次 5xx/连接失败后标记为不健康
- **恢复探测**：等待冷却期后放行一条探测请求，成功则恢复

## 配置说明

### 全局故障转移配置

```yaml
failover:
  enabled: true                    # 启用故障转移（默认 false）
  max_channel_attempts: 3          # 每个请求最多尝试几个渠道
  total_timeout_ms: 120000         # 整个请求的全局超时（毫秒）
  consecutive_fail_threshold: 2    # 5xx/连接失败连续几次后跳出渠道
  load_balance: "priority"         # 负载均衡策略：priority | round-robin | random
```

### 渠道优先级和负载均衡

```yaml
channels:
  - id: "mimo-primary"
    name: "MiMo 主力"
    priority: 10                   # 优先级（数字越小越优先，默认 100）
    # ...

  - id: "mimo-backup"
    name: "MiMo 备用"
    priority: 20
    # ...
```

**负载均衡策略（配置在 `failover.load_balance`）：**

| 策略 | 说明 |
|------|------|
| `priority` | 按优先级顺序，高优先级渠道优先使用（默认） |
| `round-robin` | 同优先级内轮转分配 |
| `random` | 同优先级内随机选择 |

## 配置示例

### 主备模式

```yaml
failover:
  enabled: true
  max_channel_attempts: 2
  total_timeout_ms: 60000
  consecutive_fail_threshold: 2
  load_balance: "priority"

channels:
  - id: "mimo-primary"
    priority: 10
    chat_url: "https://primary.example.com"
    models:
      - id: "mimo-v2.5-pro"
    keys:
      - value: "sk-key-1"
      - value: "sk-key-2"

  - id: "mimo-backup"
    priority: 20
    chat_url: "https://backup.example.com"
    models:
      - id: "mimo-v2.5-pro"
    keys:
      - value: "sk-key-3"
```

**行为：**
1. 优先使用 mimo-primary
2. 如果 primary 连续 2 次 5xx 错误 → 自动切换到 backup
3. 如果 primary 所有 Key 都返回 401 → 切换到 backup

### 负载均衡模式

```yaml
failover:
  enabled: true
  max_channel_attempts: 3
  total_timeout_ms: 120000
  load_balance: "round-robin"

channels:
  - id: "mimo-east"
    priority: 10
    chat_url: "https://east.example.com"
    models:
      - id: "mimo-v2.5-pro"
    keys:
      - value: "sk-key-1"

  - id: "mimo-south"
    priority: 10
    chat_url: "https://south.example.com"
    models:
      - id: "mimo-v2.5-pro"
    keys:
      - value: "sk-key-2"

  - id: "mimo-backup"
    priority: 20
    chat_url: "https://backup.example.com"
    models:
      - id: "mimo-v2.5-pro"
    keys:
      - value: "sk-key-3"
```

**行为：**
1. 在 east 和 south 之间轮转（同优先级 10）
2. 如果两者都不可用 → 降级到 backup（优先级 20）

## 日志和监控

故障转移事件会在日志中记录：

```
level=WARN msg=failover_next request_id=xxx failed_channel=mimo-primary reason="channel mimo-primary: 2 consecutive 503 errors" tried=1
level=WARN msg=failover_next request_id=xxx failed_channel=mimo-east reason="channel mimo-east: no available keys" tried=2
level=DEBUG msg=failover_attempt request_id=xxx channel=mimo-backup attempt=3
```

## 注意事项

1. **默认关闭**：`failover.enabled` 默认为 `false`，不配置时行为与之前完全一致
2. **向后兼容**：`priority` 默认 100，`load_balance` 默认 `priority`，单渠道场景无变化
3. **流式请求**：故障转移只在 upstream 请求阶段生效。一旦开始向客户端写入响应流，无法中途切换渠道
4. **Key 池独立**：每个渠道的 Key 池是独立的，跨渠道不会共享 Key 状态
