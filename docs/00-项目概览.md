# ai-adapter 项目概览

## 1. 项目定位

ai-adapter 是一个用 Go 编写的高性能 LLM API 协议转换代理，支持四类主流 LLM 接口协议之间的实时互转：

- **OpenAI Chat Completions** (`/v1/chat/completions`)
- **OpenAI Responses** (`/v1/responses`)
- **Anthropic Claude Messages** (`/v1/messages`)
- **Google Gemini generateContent** (`/v1beta/models/{model}:generateContent`)

```
客户端 (Codex/Claude/Gemini/任意HTTP) → ai-adapter → 渠道选择 → Key池 → 上游LLM
                                          ↓
                                    协议转换层
                            (12 个方向的请求/响应/流式转换)
```

## 2. 核心能力

| 能力 | 说明 |
|------|------|
| 四协议互转 | Chat / Responses / Claude Messages / Gemini generateContent 之间 12 方向转换 |
| 流式支持 | 所有 12 个方向同时支持 stream 和 non-stream |
| 接口能力模型 | 每个渠道通过 URL 声明原生支持的接口，未声明的接口自动选择最低复杂度转换路径 |
| 多渠道管理 | 每个渠道独立配置接口 URL、Key、模型、策略 |
| 多 Key 轮转 | round-robin / random / least-errors / least-latency 策略 |
| Key 健康追踪 | 自动暂停连续错误 Key，自动恢复 |
| Fan-out | 多 Key 并发请求，返回第一个成功结果 |
| 嵌入式 Web UI | Alpine.js 管理面板 |
| 结构化日志 | 可配置级别、请求体记录 |
| SQLite 持久化 | 请求日志和 Key 统计重启后保留 |

## 5. 接口能力模型

系统不再使用固定的渠道类型（旧 `wire_api`），而是通过四类可选 URL 声明渠道的原生接口能力：

```yaml
channels:
  - id: "example"
    chat_url: "https://api.example.com"              # 原生支持 Chat
    responses_url: "https://api.example.com"          # 原生支持 Responses
    # messages_url: "https://api.anthropic.com"       # 原生支持 Claude
    # generate_content_url: "https://..."             # 原生支持 Gemini
```

当客户端请求的接口与渠道原生接口不匹配时，系统自动选择最低复杂度的转换路径。详见 `docs/03-使用说明书.md`。
