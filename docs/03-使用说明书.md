# ai-adapter 使用说明书

## 1. 概述

ai-adapter 是一个高性能 LLM API 协议转换代理，支持四类上游接口协议之间的互转：

| 接口 | 协议 | 对外路径 |
|------|------|----------|
| `chat` | OpenAI Chat Completions | `POST /v1/chat/completions` |
| `responses` | OpenAI Responses | `POST /v1/responses` |
| `messages` | Anthropic Claude Messages | `POST /v1/messages` |
| `generate_content` | Google Gemini generateContent | `POST /v1beta/models/{model}:generateContent` |

客户端可以使用任意一种协议发起请求，系统自动选择最优的上游渠道和转换路径。

## 2. 安装

### 2.1 从源码构建

```bash
# 前置条件：Go >= 1.21
go version

# 克隆项目
git clone https://github.com/fangxiusun/ai-adapter.git
cd ai-adapter

# 构建（推荐使用构建脚本）
.\scripts\build.ps1 -Windows     # Windows
# 或
go build -o ai-adapter ./cmd/server/   # 手动构建

# 验证
./ai-adapter --help
```

### 2.2 下载预编译二进制

从 GitHub Releases 下载对应平台的二进制文件，命名格式：`ai-adapter-{os}-{arch}[.exe]`

## 3. 快速开始

### 步骤 1：创建配置文件

```bash
cp config.example.yaml config.yaml
```

### 步骤 2：编辑配置

最小可用配置（只配置一个渠道的 chat 接口）：

```yaml
server:
  port: 8080

channels:
  - id: "my-provider"
    name: "My Provider"
    enabled: true
    chat_url: "https://api.example.com"    # 该渠道原生支持 Chat Completions
    models:
      - id: "gpt-4o"
    default_model: "gpt-4o"
    keys:
      - value: "sk-your-key-here"
        name: "主 key"
```

### 步骤 3：启动

```bash
./ai-adapter -config config.yaml
```

### 步骤 4：测试

```bash
# 用 Chat Completions 协议请求（chat_url 原生支持，直接转发）
curl -X POST http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o","messages":[{"role":"user","content":"Hello"}]}'

# 用 Responses 协议请求（自动转换：responses -> chat -> 上游）
curl -X POST http://localhost:8080/v1/responses \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o","input":"Hello"}'

# 用 Claude Messages 协议请求（自动转换：messages -> chat -> 上游）
curl -X POST http://localhost:8080/v1/messages \
  -H "Content-Type: application/json" \
  -H "x-api-key: any-value" \
  -d '{"model":"gpt-4o","max_tokens":1024,"messages":[{"role":"user","content":"Hello"}]}'

# 用 Gemini 协议请求（自动转换：generate_content -> chat -> 上游）
curl -X POST http://localhost:8080/v1beta/models/gpt-4o:generateContent \
  -H "Content-Type: application/json" \
  -d '{"contents":[{"role":"user","parts":[{"text":"Hello"}]}]}'
```

所有请求到达同一个代理，系统根据目标接口和渠道能力自动选择转发或转换。

## 4. 配置详解

### 4.1 顶层结构

```yaml
server:       # 服务器配置
logging:      # 日志配置
database:     # 数据库配置
proxies:     # 代理服务器定义（可选）\nchannels:     # 渠道列表（数组）
```

### 4.2 server — 服务器配置

```yaml
server:
  host: "0.0.0.0"
  port: 8080
  api_token: ""
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `host` | string | `"0.0.0.0"` | 监听地址。`0.0.0.0` 允许外部访问，`127.0.0.1` 仅本机 |
| `port` | int | `8080` | 监听端口 |
| `api_token` | string | `""` | API 认证 Token。非空时客户端须在 `Authorization: Bearer <token>` 中携带此值。空则不认证 |
| `max_request_body_size_mb` | int | `64` | 请求体最大体积（MB），超过后截断并记录警告日志 |

### 4.3 logging — 日志配置

```yaml
logging:
  level: "info"
  file: ""
  max_size_mb: 100
  max_backups: 3
  log_request_body: false
  log_io: false
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `level` | string | `"info"` | 日志级别：`debug` / `info` / `warn` / `error` |
| `file` | string | `""` | 日志文件路径。空则输出到 stdout |
| `max_size_mb` | int | `100` | 单个日志文件最大体积（MB），超过后自动轮转 |
| `max_backups` | int | `3` | 保留的旧日志文件数量 |
| `log_request_body` | bool | `false` | 是否在 debug 日志中记录完整的请求 JSON 体。生产环境建议关闭 |
| `max_age_days` | int | `30` | 旧日志文件保留天数，超过后自动删除 |
| `compress` | bool | `true` | 是否压缩轮转后的旧日志文件 |
| `log_io` | bool | `false` | 是否记录客户端输入（`client_input`）和输出（`client_output`）。流式请求会记录每个 SSE 事件。仅在 `level: debug` 时生效 |

**日志级别说明**：

| 级别 | 输出内容 |
|------|----------|
| `debug` | 请求/响应体、工具映射、key 选择、转换路径、SSE 事件 |
| `info` | 请求摘要（路径、状态码、延迟、渠道、模型） |
| `warn` | 重试、key 暂停、429 限流 |
| `error` | 上游错误、翻译失败、key 永久跳过 |

### 4.4 database — 数据库配置

```yaml
database:
  path: "./data/proxy.db"
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `path` | string | `"./data/proxy.db"` | SQLite 数据库文件路径。存储请求日志和 Key 统计，重启后保留 |

### 4.5 proxies — 代理配置

可选配置。定义全局代理服务器，渠道通过 `proxy_id` 引用。支持 HTTP/HTTPS 代理和 SOCKS5 代理。

```yaml
proxies:
  - id: "hk-proxy"
    type: "http"
    url: "http://proxy.example.com:8080"
  - id: "us-socks"
    type: "socks5"
    url: "socks5://user:password@socks.example.com:1080"
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 代理唯一标识，渠道通过此 ID 引用 |
| `type` | string | 代理类型：`http` 或 `socks5` |
| `url` | string | 代理地址。HTTP 格式：`http://host:port`，SOCKS5 格式：`socks5://[user:pass@]host:port` |

在渠道中引用代理：

```yaml
channels:
  - id: "mimo"
    proxy_id: "hk-proxy"    # 引用上面定义的代理
    # ...其他配置
```

不设置 `proxy_id` 或留空时，该渠道直连上游，不经过代理。

### 4.6 channels — 渠道配置

渠道是系统的核心概念。每个渠道代表一个上游 LLM 提供商，通过配置接口能力 URL 声明其原生支持的协议。

```yaml
channels:
  - id: "mimo"
    name: "MiMo"
    enabled: true
    chat_url: "https://api.xiaomimimo.com"
    responses_url: "https://api.xiaomimimo.com"
    # ...其他配置
```

#### 4.6.1 基本字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 渠道唯一标识，不可重复 |
| `name` | string | 否 | 渠道显示名称，用于 Web UI |
| `enabled` | bool | 否 | 是否启用。默认 `true`。禁用后不参与路由 |
| `default_model` | string | 否 | 当请求的模型未匹配到时，回退使用的模型 ID |

#### 4.6.2 接口能力 URL

这是渠道配置的核心。设置某个 URL 即表示该渠道**原生支持**该接口协议。URL 为 base URL，**不要**包含 `/v1` 或 `/v1beta` 等路径后缀，系统会自动拼接。

| 字段 | 类型 | 说明 | 自动拼接路径 |
|------|------|------|-------------|
| `chat_url` | string | OpenAI Chat Completions 接口 | `{url}/v1/chat/completions` |
| `responses_url` | string | OpenAI Responses 接口 | `{url}/v1/responses` |
| `messages_url` | string | Anthropic Claude Messages 接口 | `{url}/v1/messages` |
| `generate_content_url` | string | Google Gemini generateContent 接口 | `{url}/v1beta/models/{model}:generateContent` |

**至少配置一个**。一个渠道可以同时配置多个接口 URL，表示同时原生支持多种协议。

**示例**：一个兼容 OpenAI 的渠道只配 `chat_url`：

```yaml
chat_url: "https://api.deepseek.com"
```

**示例**：一个同时支持 Chat 和 Responses 的渠道：

```yaml
chat_url: "https://api.xiaomimimo.com"
responses_url: "https://api.xiaomimimo.com"
```

**示例**：一个支持 Claude 原生 Messages 的渠道：

```yaml
messages_url: "https://api.anthropic.com"
```

#### 4.6.3 路由与转换规则

当客户端请求的接口类型与渠道原生支持的接口不一致时，系统自动进行协议转换：

1. **原生优先**：如果渠道配置了目标接口的 URL，直接转发（复杂度 0）
2. **转换降级**：从渠道已配置的其他接口中，选择复杂度最低的进行转换
3. **同分优先级**：`chat > responses > messages > generate_content`
4. **无可用路径**：返回 503 错误

**复杂度评分**：

| 源 \ 目标 | chat | responses | messages | generate_content |
|-----------|------|-----------|----------|-----------------|
| chat | 0 | 10 | 20 | 30 |
| responses | 10 | 0 | 20 | 30 |
| messages | 20 | 20 | 0 | 30 |
| generate_content | 30 | 30 | 30 | 0 |

分数越低越优先。同分时按固定优先级选择。

| `least-latency` | 选择平均延迟最低的 key |

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `max_retries` | int | `2` | 对单个 key 的最大重试次数 |
| `retry_delay_ms` | int | `500` | 重试间隔（毫秒） |
| `request_timeout_ms` | int | `60000` | 单次上游请求超时（毫秒） |

#### 4.6.8 fanout — 并发请求配置

```yaml
fanout:
  enabled: false

```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | `false` | 是否启用并发请求 |
| `count` | int | `2` | 并发请求数（使用 N 个 key 同时发起） |
| `wait_all` | bool | `false` | `false`：返回第一个成功结果。`true`：等待全部完成，返回延迟最低的成功结果 |

| 字段 | 类型 | 默认值 | 说明 |


| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|

#### 4.6.11 retry — 高级重试配置

```yaml
retry:
  retry_delay_429_ms: 1000
  max_rotation_rounds: 3
  max_total_wait_ms: 30000
  consec_error_threshold: 3
  pause_multiplier_sec: 30
  pause_max_sec: 600
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `retry_delay_429_ms` | int | `1000` | 遇到 429 限流时的等待时间（毫秒） |
| `max_rotation_rounds` | int | `3` | 所有 key 都返回 429 后，从头重新轮转的最大轮数 |
| `max_total_wait_ms` | int | `30000` | 整个重试过程的最大总等待时间（毫秒）。超时后将最后一个响应返回给客户端 |
| `consec_error_threshold` | int | `3` | 触发自动暂停的连续错误次数阈值 |
| `pause_multiplier_sec` | int | `30` | 暂停时间倍数。实际暂停秒数 = (连续错误数 - 阈值 + 1) x 此值 |
| `pause_max_sec` | int | `600` | 自动暂停的最大秒数（10 分钟） |

### 4.7 Header 策略配置

Header Policy Engine 允许精细控制 HTTP header 在客户端和上游服务之间的传递行为。支持三层配置（全局、渠道、模型），优先级为 `model > channel > global`。

#### 4.7.1 配置位置

Header 策略可在三个层级配置：

| 层级 | 配置字段 | 说明 |
|------|----------|------|
| 全局 | `headers.request` / `headers.response` | 顶层配置，作用于所有渠道 |
| 渠道 | `request_headers` / `response_headers` | 渠道级配置，覆盖全局 |
| 模型 | `request_headers` / `response_headers` | 模型级配置，覆盖渠道 |

#### 4.7.2 简化配置格式（推荐）

``yaml
headers:
  request:
    enabled: true
    default_action: pass           # 未匹配规则时的默认行为: pass|drop
    drop:                          # 要删除的 header
      - "X-Debug-*"                # 通配符匹配（含 *）
      - "~^x-internal-.*$"         # 正则匹配（~ 前缀）
      - "X-Remove-Me"              # 精确匹配
    pass:                          # 显式透传的 header
      - "X-Custom-Auth"
    set:                           # 设置为固定值
      "X-Gateway-ID": "ai-adapter"
    rename:                        # 重命名
      "X-Old-Name": "X-New-Name"
    append:                        # 追加值
      "X-Tags": "suffix"
    prepend:                       # 前置值
      "X-Tags": "prefix"
    copy:                          # 复制到另一个 key
      "X-Source": "X-Target"
  response:
    enabled: true
    drop: ["X-Upstream-*"]
``

#### 4.7.3 匹配方式

通过模式前缀自动识别匹配类型：

| 前缀 | 匹配方式 | 示例 | 说明 |
|------|----------|------|------|
| `~` | 正则 | `~^x-.*$` | 正则表达式匹配 |
| `*` | 通配符 | `X-Debug-*` | `*` 可在任意位置 |
| 无 | 精确 | `Content-Type` | 大小写不敏感 |

#### 4.7.4 操作类型

| 操作 | 简写语法 | 说明 |
|------|----------|------|
| drop | `drop: ["X-Debug-*"]` | 删除匹配的 header |
| pass | `pass: ["X-Custom"]` | 显式透传 |
| set | `set: {"X-Key": "value"}` | 设置为固定值 |
| rename | `rename: {"X-Old": "X-New"}` | 重命名 |
| append | `append: {"X-Key": "suffix"}` | 追加值 |
| prepend | `prepend: {"X-Key": "prefix"}` | 前置值 |
| copy | `copy: {"X-Source": "X-Target"}` | 复制到另一个 key |

#### 4.7.5 安全规则

以下 header 在 **request phase** 会被强制 drop，不可被任何规则覆盖：

- `Authorization`（精确匹配）
- `Cookie`（精确匹配）
- `X-Internal-*`（通配符匹配）

> **注意**：upstream 请求的 `Authorization` 由 key pool 单独设置（`Bearer {key}`），不受策略引擎影响。

#### 4.7.6 完整规则格式（高级）

对于需要更精细控制的场景，可以使用完整规则格式：

``yaml
headers:
  request:
    enabled: true
    default_action: pass
    rules:
      - name: "custom-rule"
        phase: request
        match_type: exact
        pattern: "X-Custom-Header"
        action: set
        value: "custom-value"
``

#### 4.7.7 配置示例

``yaml
# 全局配置（简化格式）
headers:
  request:
    enabled: true
    default_action: pass
    drop: ["X-Debug-*", "~^x-internal-.*$"]
    set:
      "X-Gateway-ID": "ai-adapter"
  response:
    enabled: true
    drop: ["X-Upstream-*"]

# 渠道级配置（覆盖全局）
channels:
  - id: "mimo"
    request_headers:
      enabled: true
      drop: ["X-Channel-Debug-*"]
      set:
        "X-Channel-ID": "mimo"
``

## 8. 错误处理

### 8.1 错误响应格式

```json
{
  "error": {
    "type": "error",
    "code": "no_channel",
    "message": "no channel found for model: xxx",
    "status": 404
  }
}
```

### 8.2 常见错误代码

| HTTP 码 | 错误代码 | 说明 | 解决方案 |
|---------|----------|------|----------|
| 400 | `invalid_json` | 请求体 JSON 解析失败 | 检查请求体格式 |
| 400 | `missing_model` | 请求缺少 model 字段 | 在请求体或 URL 中指定 model |
| 400 | `convert_failed` | 协议转换失败 | 检查请求体是否符合所用协议规范 |
| 404 | `no_channel` | 未找到匹配渠道 | 检查 model ID 和渠道 models 配置 |
| 502 | `upstream_error` | 上游返回错误 | 检查 API Key 和上游服务状态 |
| 503 | `no_available_keys` | 所有 Key 都不可用 | 可能全部被 401 暂停，需在 Web UI 中 Resume |
| 503 | `no_conversion_path` | 渠道无可用转换路径 | 渠道未配置任何接口 URL，或缺少可转换的源接口 |
| 504 | `timeout` | 重试总超时 | 检查上游响应速度，调整 `max_total_wait_ms` |

### 8.3 自动重试行为

| 上游错误 | 处理方式 |
|----------|----------|
| 400 Bad Request | 跳过当前 Key，切换下一个 |
| 401 Unauthorized | 永久跳过该 Key，切换下一个 |
| 403 Forbidden | 跳过当前 Key，切换下一个 |
| 404 Not Found | 跳过当前 Key，切换下一个 |
| 429 Rate Limited | 跳过当前 Key，等待后切换 |
| 其他 4xx | 跳过当前 Key，切换下一个 |
| 5xx Server Error | 跳过当前 Key，切换下一个 |
| 网络错误 | 跳过当前 Key，切换下一个 |

## 9. 故障排查

### 9.1 启动失败

**`load config: open config.yaml: no such file`**

```bash
ls -la config.yaml
./ai-adapter -config /absolute/path/to/config.yaml
```

**`channel xxx: wire_api has been removed`**

旧配置格式已废弃。将 `wire_api: "chat"` 替换为 `chat_url: "https://..."`，将 `wire_api: "responses"` 替换为 `responses_url: "https://..."`。

**`channel xxx: at least one interface capability URL is required`**

每个渠道必须至少配置一个接口 URL（`chat_url` / `responses_url` / `messages_url` / `generate_content_url`）。

### 9.2 请求失败

**`no_conversion_path`**

渠道没有配置可转换到目标接口的 URL。例如客户端发 `/v1/messages` 请求，但渠道只配了 `chat_url`，而 `messages -> chat` 转换是支持的，应正常工作。如果报此错，检查渠道是否至少有一个接口 URL。

**`all_retries_failed` 或 401**

```bash
# 检查 Key 是否有效
curl https://api.example.com/v1/models -H "Authorization: Bearer sk-your-key"

# 在 Web UI 中 Resume 被暂停的 Key
# 或重启代理重置所有 Key 状态
```

**频繁 429**

增加 Key 数量分散压力，或使用 `least-latency` 策略：

```yaml
key_strategy: "least-latency"
keys:
  - value: "sk-key-1"
  - value: "sk-key-2"
  - value: "sk-key-3"
```

### 9.3 查看详细日志

```yaml
logging:
  level: "debug"
  log_request_body: true
```

重启代理后查看 stdout 或日志文件中的详细请求/响应信息。

### 9.4 后台运行模式（Daemon）

```bash
./ai-adapter -config config.yaml -daemon
```

启动后进程在后台运行，终端立即释放。日志输出到配置的 `logging.file` 路径，未配置时使用可执行文件同目录下的 `ai-adapter.log`。

**Windows 注意事项**：daemon 模式使用 `CREATE_NO_WINDOW` 标志创建隐藏进程，不会弹出控制台窗口。

**停止后台进程**：

```bash
# Windows
Get-Process | Where-Object { $_.ProcessName -like '*ai-adapter*' } | Stop-Process

# Linux/macOS
pkill ai-adapter
```

### 9.5 深度调试模式

当需要排查请求在 adapter 内部的完整流转路径时，可启用深度调试模式：

```bash
# 方式1: 命令行参数
./ai-adapter --deep-debug -config config.yaml

# 方式2: 环境变量
AI_ADAPTER_DEEP_DEBUG=true ./ai-adapter -config config.yaml
```

每个请求会在 `./debug_logs/` 下生成一个独立目录，目录名为 LOGID（格式：`YYYYMMDDHHMMSSmmm-UUID`），包含以下 8 个文件：

| 文件 | 内容 |
|------|------|
| `{LOGID}.client.req.header.log` | 客户端请求 headers + URL |
| `{LOGID}.client.req.body.log` | 客户端请求 body（JSON 自动格式化） |
| `{LOGID}.upstream.req.header.log` | 发送给 upstream 的 headers（token 自动脱敏） |
| `{LOGID}.upstream.req.body.log` | 发送给 upstream 的 body（JSON 自动格式化） |
| `{LOGID}.upstream.res.header.log` | upstream 返回的 headers + 状态码 |
| `{LOGID}.upstream.res.body.log` | upstream 返回的 content（JSON 自动格式化） |
| `{LOGID}.client.res.header.log` | 返回给客户端的 headers |
| `{LOGID}.client.res.body.log` | 返回给客户端的 content（JSON 自动格式化） |

通过对比 `client.req.body.log` 和 `upstream.req.body.log` 可以确认协议转换是否正确；对比 `client.req.header.log` 和 `upstream.req.header.log` 可以确认请求头的转发情况。



## 10. 跨渠道故障转移

当多个渠道配置了相同模型时，可以启用故障转移功能，自动在渠道间切换以提高可用性。

### 快速配置

```yaml
failover:
  enabled: true
  max_channel_attempts: 3
  total_timeout_ms: 120000
  consecutive_fail_threshold: 2

channels:
  - id: "primary"
    priority: 10
    chat_url: "https://primary.example.com"
    # ...
  - id: "backup"
    priority: 20
    chat_url: "https://backup.example.com"
    # ...
```

### 工作原理

- **401/429/4xx**：先在渠道内轮转 Key，所有 Key 用尽后切换渠道
- **5xx/连接失败**：连续 N 次失败后立即切换渠道（不浪费 Key 轮转次数）
- **400**：直接返回错误，不做重试

### 渠道优先级

- `priority` 数字越小优先级越高（默认 100）
- 同优先级内支持 `round-robin` 轮转和 `random` 随机策略
- 高优先级渠道全部不可用时自动降级到低优先级渠道

详细配置说明请参阅 [跨渠道故障转移与负载均衡指南](11-跨渠道故障转移与负载均衡.md)。
