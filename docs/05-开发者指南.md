## 2. 开发环境搭建

### 2.1 前置条件

- Go >= 1.21
- Git

### 2.2 克隆与构建

```bash
git clone https://github.com/fangxiusun/ai-adapter.git
cd ai-adapter

# 构建
go build -o ai-adapter ./cmd/server/

# 运行测试
go test ./...

# 开发模式
go run ./cmd/server/ -config config.yaml
```

### 2.3 使用构建脚本

```bash
# Windows PowerShell
.\scripts\build.ps1              # 构建当前平台
.\scripts\build.ps1 -All         # 构建所有平台
.\scripts\build.ps1 -Windows     # 仅 Windows

# Linux / macOS
./scripts/build.sh               # 构建所有平台
./scripts/build.sh linux         # 仅 Linux
```

构建产物输出到 `dist/`，自动注入 git 版本号。

### 2.4 依赖管理

```bash
go get github.com/example/pkg
go mod tidy
go mod graph
```

## 3. 代码规范

### 3.1 包组织

- `cmd/`：入口程序
- `internal/`：内部包（不可被外部导入）
- 每个包职责单一：`translate/` 仅做协议翻译，`channel/` 仅做渠道和 Key 管理，`proxy/` 仅做请求路由和协调

### 3.2 命名规范

- 包名：小写单词（`translate`, `channel`）
- 文件名：小写 + 下划线（`keypool.go`, `claude_convert.go`）
- 函数名：PascalCase（`ChatToClaudeRequest`, `BestSourceForTarget`）
- 变量名：camelCase（`upstreamModel`, `chatReq`）

### 3.3 错误处理

```go
func doSomething() error {
    if err := validate(); err != nil {
        return fmt.Errorf("validate: %w", err)
    }
    return nil
}
```

## 4. 核心概念

### 4.1 接口能力模型

系统通过四类接口 URL 声明渠道的原生协议支持能力。类型定义在 `internal/config/capability.go`：

```go
type InterfaceType string

const (
    InterfaceChat            InterfaceType = "chat"
    InterfaceResponses       InterfaceType = "responses"
    InterfaceMessages        InterfaceType = "messages"
    InterfaceGenerateContent InterfaceType = "generate_content"
)
```

路由决策函数 `BestSourceForTarget` 根据复杂度评分表选择最优源接口。

### 4.2 协议转换架构

所有转换以 Chat 格式为 hub（中心）：

```
Responses <---> Chat <---> Claude Messages
                  ^
                  |
          Gemini generateContent
```

- Chat <-> Responses：`translate/req.go`, `translate/resp.go`, `translate/stream.go`
- Chat <-> Claude：`translate/claude_convert.go`
- Chat <-> Gemini：`translate/gemini_convert.go`
- 其他方向（如 Responses -> Claude）：链式通过 Chat 中转

### 4.3 Key 管理

Key 池支持五种轮转策略：

```go
// round-robin：轮询（默认）
// random：随机
// least-errors：最少错误
// least-latency：最低延迟
// least-rate-limited：最少限流
```

**Key 统计持久化**：Key 池启动时从 SQLite 加载历史统计数据（`loadFromDB`），运行期间通过后台 goroutine 定期同步到数据库（`syncLoop`），同步间隔由配置项 `key_stats_sync_sec` 控制（默认 60 秒）。关闭时执行一次最终同步（`SaveToDB`）。

**错误分类统计**：每个 Key 独立统计以下错误类型：`error_400`、`error_401`、`error_403`、`error_404`、`error_429`、`error_4xx`、`error_5xx`、`error_network`、`error_stream`。这些统计数据用于 least-errors 和 least-rate-limited 策略的选择决策。

### 4.4 流式翻译

流式翻译使用状态机模式。Chat <-> Responses 的流式翻译在 `translate/stream.go`，使用 `StreamTranslator` 状态机。Claude 和 Gemini 的流式转换在各自的 `*_convert.go` 文件中，通过 SSE 事件逐个转换。

### 4.5 代理处理器

`proxy/handler.go` 是系统核心，包含四个入口函数：

```go
func (h *ProxyHandler) HandleChat(w, r)            // POST /v1/chat/completions
func (h *ProxyHandler) HandleResponses(w, r)        // POST /v1/responses
func (h *ProxyHandler) HandleMessages(w, r)         // POST /v1/messages
func (h *ProxyHandler) HandleGenerateContent(w, r)  // POST /v1beta/models/...
```

统一的 `dispatch` 函数负责：
1. 调用 `BestSourceForTarget` 选择源接口
2. 原生接口 → 直接转发
3. 非原生 → 通过 Chat hub 转换后转发
4. 流式 → 使用事件级管道或累积后重放
5. 请求完成后调用 `RecordLatency` 记录 Key 的延迟统计，用于 least-latency 策略

深度调试日志（`internal/debuglog/`）在每个入口函数和转发函数中注入日志点：
- `HandleXxx`：记录 client request header/body
- `nativeForward` / `convertedNonStreamForward` / `streamFromChatSource` / `streamChainConversion`：记录 upstream request 和 response 的 header/body

### 4.6 后台运行模式 (`internal/daemon/`)

`daemon.go` 实现跨平台后台运行：

- **Unix**：`fork` 子进程，stdin/stdout/stderr 重定向到 `/dev/null` 或日志文件
- **Windows**：使用 `CREATE_NO_WINDOW` 标志创建隐藏进程，不占用终端
- 子进程通过 `AI_ADAPTER_DAEMON=1` 环境变量避免重复 fork
- 子进程启动后将 `os.Stdin` 替换为 `os.DevNull`，确保不持有父终端

### 4.7 Header 策略引擎 (`internal/headerpolicy/`)

`engine.go` 实现请求/响应 header 的精细控制：

- 支持 exact、wildcard (`*`)、regex (`~` 前缀) 三种匹配模式
- 操作：drop / set / pass / rename / append / prepend / copy
- 安全规则：`Authorization`、`Cookie`、`X-Internal-*` 在请求阶段始终被丢弃
- 支持全局和渠道级配置，渠道级覆盖全局

### 4.8 Prometheus Metrics (`internal/metrics/`)

`metrics.go` 定义 Prometheus 指标：

- `http_requests_total`：按 method、path、status 分类的请求计数
- `http_request_duration_seconds`：请求延迟直方图
- `active_requests`：当前活跃请求数
- `key_errors_total`：按 channel、key、error_type 分类的 Key 错误计数

### 4.9 WebSocket 实时推送 (`internal/websocket/`)

`hub.go` 实现 WebSocket 连接管理和消息广播：

- 管理面板通过 WebSocket 接收实时 QPS、延迟、错误率数据
- 心跳机制每 30 秒发送一次，包含当前活跃请求数

### 4.10 负载均衡 (`internal/channel/balancer.go`)

`balancer.go` 定义 `Balancer` 接口和三种实现：

- `PriorityBalancer`：按优先级分组，高优先级组内轮询
- `RoundRobinBalancer`：全局轮询
- `RandomBalancer`：随机选择

### 4.11 渠道健康追踪 (`internal/channel/health.go`)

`health.go` 跟踪每个渠道的健康状态：

- 连续失败超过阈值后标记为不健康
- 冷却期后自动恢复

## 5. 添加新功能

### 5.1 添加新的协议转换

1. 在 `internal/translate/` 创建 `xxx_types.go` 定义协议类型
2. 创建 `xxx_convert.go` 实现 Chat <-> XXX 双向转换（请求、响应、流式）
3. 在 `proxy/convert.go` 的 `convertChatToSource` / `convertSourceToChat` / `convertChatToTarget` 中添加 case
4. 在 `proxy/handler.go` 的 `buildChatRequest` 和 `accumulateStreamToChat` 中添加 case
5. 在 `config/capability.go` 的 `InterfaceType`、`AllInterfaces`、复杂度表中添加新类型
6. 在 `cmd/server/main.go` 注册新路由

### 5.2 添加新的渠道特性

1. 在 `config.go` 的 `ChannelConfig` 中添加字段
2. 在 `channel.go` 中使用新字段
3. 在 `config.example.yaml` 中添加示例

### 5.3 添加新的 Admin API

1. 在 `web/handler.go` 中添加处理函数
2. 在 `RegisterRoutes` 中注册路由
3. 在 Web UI 中添加对应页面
