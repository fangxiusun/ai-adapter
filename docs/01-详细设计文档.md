## 2. 核心模块设计

### 2.1 协议翻译层 (`internal/translate/`)

这是系统的核心——在 Responses API 和 Chat Completions API 之间做实时翻译。

#### 2.1.1 类型定义 (`types.go`)

定义两套 API 的 Go 结构体：

**Responses API 核心类型**：
- `ResponsesRequest`：请求体（model, input, instructions, tools, reasoning...）
- `ResponsesInputItem`：输入项（message/function_call/function_call_output/reasoning）
- `ResponsesObject`：响应对象（output, usage, status...）
- `OutputItem`：输出项（message/function_call/reasoning）

**Chat Completions API 核心类型**：
- `ChatRequest`：请求体（model, messages, tools, thinking...）
- `ChatMessage`：消息（role, content, tool_calls, reasoning_content...）
- `ChatResponse`：响应（choices, usage...）
- `ChatStreamChunk`：流式块

#### 2.1.2 请求翻译 (`req.go`)

两个核心函数：

```go
// Responses → Chat
func ReqToChat(req *ResponsesRequest, opts TranslateOpts) (*ChatRequest, error)

// Chat → Responses
func ReqToResponses(req *ChatRequest, opts TranslateOpts) (*ResponsesRequest, error)
```

**ReqToChat 翻译逻辑**：

```
ResponsesRequest
    │
    ├── instructions → messages[0] (role: "system")
    │
    ├── input (string) → messages[N] (role: "user", content: string)
    │
    ├── input (array) → inputItemsToMessages()
    │   ├── message item → message (role 直接映射)
    │   ├── reasoning item → 折叠到 assistant.reasoning_content
    │   ├── function_call item → 折叠到 assistant.tool_calls
    │   └── function_call_output → tool message
    │
    ├── tools[] → ToolToChat() × N → ChatTool[]
    │
    ├── tool_choice → ToolChoiceToChat()
    │
    ├── reasoning.effort → reasoning_effort
    │
    └── 其他参数直传
```

**关键设计**：同一回合的 reasoning + assistant text + function_call 合并为一条 assistant message（状态机模式）。

**消息完整性保障**：
- `removeOrphanToolMessages()`：删除孤立 tool 消息
- `ensureToolCallsHaveOutputs()`：补充缺失的 tool 输出

#### 2.1.3 响应翻译 (`resp.go`)

```go
// Chat → Responses（非流式）
func RespToResponses(resp *ChatResponse, req *ResponsesRequest, opts TranslateOpts) *ResponsesObject

// Responses → Chat（非流式）
func RespToChat(resp *ResponsesObject, req *ChatRequest, opts TranslateOpts) *ChatResponse
```

**RespToResponses 逻辑**：
1. `reasoning_content` → `ResponsesReasoningItem`（`encrypted_content` 始终填充）
2. `content` → `ResponsesMessageItem`
3. `tool_calls[]` → `ResponsesFunctionCallItem[]`
4. `usage` 字段映射

#### 2.1.4 流式翻译 (`stream.go`)

```go
// Chat SSE → Responses SSE
func PipeChatStreamToResponses(ctx, upstream, sink, req, opts) (*StreamResult, error)

// Responses SSE → Chat SSE
func PipeResponsesStreamToChat(ctx, upstream, sink, req, opts) (*ChatResponse, error)
```

**流式状态机**：
- `activeKind`: "reasoning" | "message" | null
- `activeBuffer`: 文本缓冲
- `toolCalls`: Map[index, ToolCallState]
- 每个 item 遵循：open → delta* → finalize

**SSE 事件序列**：
```
response.created → response.in_progress →
  [response.output_item.added → (delta)* → response.output_item.done] × N →
response.completed
```

#### 2.1.5 工具映射 (`tools.go`)

| Responses 类型 | Chat 类型 | 处理 |
|---------------|-----------|------|
| `function` | `function` | 直接透传 |
| `local_shell` | `function` (name="shell") | 转换 |
| `web_search` | `web_search` | 条件转发 |
| `custom` | `function` | 转换 |
| `namespace` | (递归展开) | 展开内部工具 |
| `mcp` | (丢弃) | 注入 advisory note |
| `tool_search` | `function` | 转换 |
| 其他内置 | (丢弃) | 静默丢弃 |

#### 2.2.2 Key 池 (`keypool.go`)

```go
type KeyPool struct {
    keys         []*KeyEntry
    strategy     string  // round-robin | random | least-errors | least-latency | least-rate-limited
    counter      uint64
    database     *db.DB  // 可选，用于持久化 key 统计
    syncInterval time.Duration  // 同步间隔（key_stats_sync_sec 配置）
}

func (kp *KeyPool) Next() *KeyEntry                    // 获取下一个可用 key
func (kp *KeyPool) NextExcluding(exclude map[string]bool) *KeyEntry  // 排除指定 key 后选择
func (kp *KeyPool) GetN(n int) []*KeyEntry              // 获取 N 个 key（fan-out 用）
func (kp *KeyPool) ReportSuccess(key string)
func (kp *KeyPool) ReportError(key string, statusCode int)  // 根据状态码分发处理
func (kp *KeyPool) ReportStreamError(key string)        // 流式解析错误
func (kp *KeyPool) RecordLatency(key string, ms int64)  // 记录请求延迟
func (kp *KeyPool) SaveToDB()                           // 手动触发同步到数据库
func (kp *KeyPool) Stop()                               // 停止后台同步
```

**轮转策略**：
- `round-robin`：轮询（默认）
- `random`：随机选择
- `least-errors`：选择错误最少的 key
- `least-latency`：选择平均延迟最低的 key
- `least-rate-limited`：选择近期被 429 限制最少的 key（按 429 频率排序）

**ReportError 状态码分发**：
- **400**：调用 `On400()` 记录错误，连续错误触发暂停
- **401**：调用 `On401()` 永久跳过该 key，日志明文打印 key 名称和值
- **403**：调用 `On403()` 记录错误，连续错误触发暂停
- **404**：调用 `On404()` 记录错误，连续错误触发暂停
- **429**：调用 `On429()` 记录速率限制，暂停 30s~10min
- **其他 4xx**：调用 `OnError4xx()` 记录错误
- **5xx**：调用 `OnError5xx()` 记录错误，连续错误触发暂停
- **网络错误**：调用 `OnErrorNetwork()` 记录错误

**持久化机制**：启动时从 SQLite 加载历史统计数据（`loadFromDB`），运行期间通过后台 goroutine 定期同步（`syncLoop`），间隔由 `key_stats_sync_sec` 配置（默认 60 秒）。

#### 2.2.3 Key 状态 (`keystate.go`)

```go
type KeyState struct {
    RequestCount       int64
    ErrorCount         int64
    Error400           int64  // 400 错误计数
    Error401           int64  // 401 错误计数
    Error403           int64  // 403 错误计数
    Error404           int64  // 404 错误计数
    Error429           int64  // 429 错误计数
    Error4xx           int64  // 其他 4xx 错误计数
    Error5xx           int64  // 5xx 错误计数
    ErrorNetwork       int64  // 网络错误计数
    ErrorStream        int64  // 流式解析错误计数
    TotalLatencyMs     int64  // 总延迟（用于计算平均延迟）
    ConsecErrors       int
    Paused             bool
    PauseUntil         time.Time
    PermanentlySkipped bool   // 401 永久跳过
    RateLimitCount     int    // 5分钟窗口内 429 次数
    RateLimitWindow    []time.Time  // 429 时间窗口
}

func (ks *KeyState) IsAvailable() bool
func (ks *KeyState) On400()              // 400 错误
func (ks *KeyState) On401()              // 永久跳过（401 Unauthorized）
func (ks *KeyState) On403()              // 403 错误
func (ks *KeyState) On404()              // 404 错误
func (ks *KeyState) On429()              // 速率限制（429 Rate Limited）
func (ks *KeyState) OnError4xx()         // 其他 4xx 错误
func (ks *KeyState) OnError5xx(status int) // 5xx 错误
func (ks *KeyState) OnErrorNetwork()     // 网络错误
func (ks *KeyState) OnErrorStream()      // 流式解析错误
func (ks *KeyState) OnSuccess()
func (ks *KeyState) RecordLatency(ms int64) // 记录延迟
func (ks *KeyState) AvgLatencyMs() float64  // 返回平均延迟
func (ks *KeyState) RateLimitScore() int  // 返回近期 429 次数（用于排序）
```

**401 处理**：
- `On401()` 设置 `PermanentlySkipped = true`
- 该 key 永久不可用，除非手动 Resume

**429 处理**：
- `On429()` 维护 5 分钟滑动窗口
- 连续错误 ≥3 次 → 暂停 30s~10min（指数退避）
- `RateLimitScore()` 返回窗口内 429 次数，用于 `least-rate-limited` 策略排序

**通用错误暂停策略**：
- 连续错误 ≥ `consecThreshold`（默认 5）→ 暂停
- 暂停时间 = `(consecErrors - threshold + 1) × 60秒`，最长 30 分钟
- 恢复成功后重置连续错误计数

#### 2.2.4 Fan-out (`fanout.go`)

```go
func (ch *Channel) Fanout(ctx context.Context, req FanoutRequest) *FanoutResult
```

**流程**：
1. 从 KeyPool 获取 N 个不同的 key
2. 并发发送请求到每个 key
3. `wait_all=false`：第一个成功响应立即返回，取消其余
4. `wait_all=true`：等待所有完成，选择延迟最低的成功响应

### 2.3 代理处理器 (`internal/proxy/`)

```go
type ProxyHandler struct {
    channels *channel.ChannelManager
    db       *db.DB
    logger   *applog.Logger
}

func (h *ProxyHandler) HandleResponses(w, r)  // POST /v1/responses
func (h *ProxyHandler) HandleChat(w, r)       // POST /v1/chat/completions
```

**请求处理流程（含重试）**：

```
1. 读取请求体
2. 提取 model 字段
3. channel.SelectChannel(model) 选择渠道
4. 检查渠道 interface capability URLs：
   - 配置了 chat_url 且请求是 Responses → ReqToChat() 翻译
   - 配置了 responses_url 且请求是 Chat → ReqToResponses() 翻译
   - 格式匹配 → 直接透传
5. 进入重试循环：
   for attempt := 0; attempt <= maxRetries; attempt++ {
     a. 从 KeyPool 获取下一个可用 key（排除已失败的）
     b. 发送上游请求
     c. 如果 401：
        - 永久跳过该 key（日志明文打印 key 名称和值）
        - 继续下一个 key
     d. 如果 429：
        - 跳过该 key
        - 等待 500ms × (attempt+1) 后重试
        - 继续下一个 key
     e. 如果 5xx：
        - 跳过该 key
        - 继续下一个 key
     f. 如果成功：
        - 响应翻译回客户端格式
        - 记录日志
        - 返回响应
   }
6. 所有重试耗尽 → 返回 503 错误
```

**重试策略**：
- **401 错误**：永久跳过该 key，不重试同一 key，切换到下一个 key
- **429 错误**：跳过当前 key，等待递增延迟后切换到下一个 key
- **5xx 错误**：跳过当前 key，切换到下一个 key
- **所有 key 都失败**：返回 503 错误给客户端

**延迟策略**：
- 429 重试延迟 = `500ms × (attempt + 1)`（递增）
- 例如：第1次等待500ms，第2次等待1000ms，第3次等待1500ms

## 3. 数据流详解

### 3.1 Responses → Chat 流程

```
1. 客户端发送 POST /v1/responses
   Body: { model: "mimo-v2.5-pro", input: [...], tools: [...] }

2. 解析请求体

3. 选择渠道：SelectChannel("mimo-v2.5-pro")
   → 匹配 mimo 渠道，配置了 chat_url

4. 请求翻译：ReqToChat()
   - instructions → messages[0] (system)
   - input[] → messages[1:] (user/assistant/tool)
   - tools[] → ChatTool[] (function 映射)
   - reasoning.effort → reasoning_effort

5. 获取 Key：keyPool.Next()
   → round-robin 选择下一个可用 key

6. 发送上游请求
   POST https://api.xiaomimimo.com/v1/chat/completions
   Authorization: Bearer sk-xxx
   Body: { model: "mimo-v2.5-pro", messages: [...], tools: [...] }

7. 上游返回 ChatResponse

8. 响应翻译：RespToResponses()
   - choices[0].message.reasoning_content → reasoning item
   - choices[0].message.content → message item
   - choices[0].message.tool_calls[] → function_call items
   - usage → ResponsesUsage

9. 返回 ResponsesObject 给客户端
```

### 3.2 Chat → Responses 流程

```
1. 客户端发送 POST /v1/chat/completions
   Body: { model: "deepseek-v4-pro", messages: [...] }

2. 选择渠道：SelectChannel("deepseek-v4-pro")
   → 匹配 deepseek 渠道，配置了 chat_url

3. 因为 配置了 chat_url 且请求是 Chat → 直接透传（无需翻译）

4. 发送上游请求
   POST https://api.deepseek.com/v1/chat/completions
   Body: { model: "deepseek-v4-pro", messages: [...] }

5. 上游返回 ChatResponse

6. 直接返回给客户端
```

### 3.3 流式翻译流程

```
1. 客户端发送 POST /v1/responses (stream: true)

2. 翻译为 ChatRequest

3. 发送 HTTP 200 + SSE 头

4. 调用上游（流式）

5. StreamTranslator 处理每个 chunk：
   - reasoning_content → openReasoning() + delta events
   - content → openMessage() + delta events
   - tool_calls → openToolCall() + delta events

6. 流结束：
   - finalizeActive() → 关闭当前 item
   - finalizeToolCalls() → 关闭所有工具调用
   - emit response.completed

7. 记录日志
```

- 至少一个渠道

- 每个渠道必须有 chat_url/responses_url/messages_url/generate_content_url、keys、models
- 每个渠道至少配置一个接口能力 URL
